# 07-工程风险与实施策略

## 1. 总体判断

本项目可行，但风险不在于“代码写不出来”，而在于长期维护、数据质量、状态复杂度和流程边界。

如果一次性追求全自动，容易变成半成品。应采用渐进明细原则，先做最小闭环，再逐步扩展。

## 2. 对比纯人工工作量

对比人工从 Google Scholar 搜索、下载、整理 Excel、写报告，系统能节省的主要不是下载 PDF 的点击次数，而是：

- 发现；
- 去重；
- 记录；
- 筛选；
- 状态维护；
- 证据句定位；
- 报告整理。

预估节省：

```text
第一阶段：40%～60%
第二阶段：60%～75%
第三阶段：75%～85%
```

适用判断：

```text
少量临时查询：手动 Scholar 更快
每月几十篇：系统开始有价值
每月 100 篇以上：系统明显节省工作量
需要长期报告：系统价值远高于手工
需要多人协作：必须系统化
```

## 3. 数据源风险

不同数据源字段不一致：

- DOI 缺失或格式不同；
- 标题有副标题、大小写和符号差异；
- 作者顺序不一致；
- 摘要有的源有、有的源没有；
- online date 和 print date 不一致。

应对策略：

- 去重模块作为核心模块；
- 保留 `paper_sources` 表；
- 主键优先级：DOI > PMID > PMCID > 标准化标题 + 第一作者 + 年份；
- 不丢弃来源记录。

## 4. 状态复杂度风险

一篇论文可能同时具有多个状态：

```text
已获取摘要
全文受限
手动上传全文
解析失败
需要人工审核
参考价值未知
```

应对策略：

- 不使用单一 status；
- 使用多个状态字段；
- 自动生成用户可见标签。

关键字段：

- abstract_status；
- fulltext_status；
- parse_status；
- analysis_status；
- evidence_level；
- reference_value；
- review_status；
- manual_override。

## 5. PDF 解析风险

PDF 解析常见问题：

- 分栏导致文本顺序错乱；
- Methods 章节被拆散；
- 表格、图注、补充材料提取不完整；
- 扫描版 PDF 没有文本层；
- 希腊字母、上下标、基因名、特殊符号被解析错；
- genOway 可能出现在脚注、作者单位、利益冲突里，不一定代表使用了模型。

应对策略：

- 第一版不追求完美解析；
- 重点提取大段文本、Methods、Funding、Conflict；
- 所有自动结论保留证据句；
- 不确定结果进入人工审核。

## 6. 全文获取风险

即使有 DOI，也不是每篇都能合法下载全文。

状态应区分：

```text
unavailable = 没找到开放全文
restricted = 疑似受限
failed = 技术失败
manual_needed = 建议人工补充
```

拿不到全文是正常状态，不是失败。

## 7. genOway 相关性判断风险

命中 genOway 不代表论文使用了 genOway 模型。命中模型名也不一定代表与 genOway 有明确商业关系。

应对策略：

证据等级必须保守：

```text
A：明确写模型来自 genOway / 基锘威
B：全文提到 genOway，但模型关系需确认
C：只在单位、资助、利益冲突中出现
D：只命中模型名或关键词
X：明确无关
```

第一版不要自动把论文标成“可用于市场材料”。最多自动建议高价值候选，最终由人工确认。

## 8. LLM 抽取风险

如果使用 LLM 抽取模型、靶点、疾病、应用场景，可能产生幻觉。

风险包括：

- 把疾病背景当成实验疾病；
- 把讨论中的模型当成实际使用模型；
- 把供应商名字误认为合作方；
- 把作者单位误认为客户；
- 把摘要里的泛化描述当成明确证据。

应对策略：

所有抽取结果必须包含：

```text
字段 = 抽取结果
evidence_sentence = 原文证据句
section_name = 证据所在章节
confidence_score = 置信度
needs_review = true/false
```

没有证据句的结构化结果不允许进入高价值报告。

## 9. 手动导入匹配风险

用户手动上传 PDF 后，系统要判断它对应哪篇论文。

风险包括：

- PDF 文件名不规范；
- PDF 元数据没有 DOI；
- PDF 首页解析失败；
- preprint 和正式发表版标题不同；
- 一篇论文多个版本被重复导入。

应对策略：

```text
高置信匹配：自动关联
中置信匹配：提示候选 paper_id
低置信匹配：创建 manual_only 记录，进入人工审核
```

## 10. WebUI 风险

WebUI 容易变成复杂系统，拖慢第一版开发。

应对策略：

- 第一版先不做 WebUI；
- 第二阶段用 Streamlit；
- 不做复杂权限；
- 不在 WebUI 页面直接拼 SQL；
- WebUI 只做审核和操作入口，不做大规模批处理；
- 大规模任务仍由 CLI 执行。

## 11. 中文数据源风险

CNKI、万方、维普等中文库工程化难度高。

风险包括：

- 没有稳定开放 API；
- 订阅限制；
- CAJ 格式；
- 自动抓取合规风险；
- 元数据质量不统一。

应对策略：

- 第一版只做人工入口；
- 支持题录导入；
- 支持中文 PDF；
- CAJ 只登记待转换；
- 中文线索默认 needs_review。

## 12. 幂等性和人工覆盖风险

系统会反复运行，如果幂等性不好，数据库会变乱。

必须保证：

- 同一检索重复运行不会重复入库；
- 同一 PDF 不会重复下载；
- 同一全文不会重复解析，除非 `--force`；
- 同一证据句不会重复插入；
- 中途失败后可以继续；
- 人工确认过的字段不会被自动任务覆盖。

建议：

```text
manual_override = true
```

自动任务遇到人工确认字段时不得覆盖。

## 13. 报告风险

报告容易“信息很多，但不能用”。

建议分层：

管理层报告：

- 本周新增高价值论文；
- 重点模型应用变化；
- 重点疾病领域；
- 建议跟进方向。

销售/市场报告：

- 可用于案例材料的论文；
- 证据句；
- 模型名称；
- 疾病领域；
- 应用场景；
- 注意事项。

维护人员报告：

- 待审核；
- 全文受限；
- 解析失败；
- 重复记录；
- 低置信结果；
- CAJ 待转换。

## 14. Codex 执行策略

不要让 Codex 一次性全做。

推荐里程碑：

第一步：

```text
SQLite + PubMed + Crossref + 去重 + CLI + Excel 导出
```

第二步：

```text
Unpaywall + PDF 下载 + 手动 PDF 导入
```

第三步：

```text
PDF 解析 + 证据抽取 + 状态机
```

第四步：

```text
Streamlit WebUI + 外部链接入口 + 手动审核
```

第五步：

```text
中文数据源入口 + CAJ 登记 + 中国境内线索报告
```

每一步要求：

```text
能运行
有测试
有示例数据
有 README
```

## 15. 最小可用闭环

第一版应先跑通：

```text
PubMed 摘要获取
→ DOI 补全
→ 手动 PDF 导入
→ PDF 解析
→ 证据句抽取
→ 人工标记高/低价值
→ Excel 报告
```

这个闭环跑通后，再加自动全文获取、WebUI 和中文数据源。
